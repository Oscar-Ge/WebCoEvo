\section{Results}

\input{figures/latex_includes.tex}

\paragraph{Experimental setting.}
We evaluate a fixed multimodal web agent on a self-hosted bookmark-management website under three conditions: no retrieved rules, \expel{}-style rules~\citep{zhao2023expel}, and \expel{}-style rules augmented with reflection rules. The evaluation uses 68 training tasks for diagnosing rule behavior and 167 held-out validation tasks for testing transfer to unseen task families. We report task success rate on \linkding{} V1.45.0, on a moderately modified website V2, and on a harder generated website suite V3 that changes access, visual presentation, content, timing, workflow, navigation structure, and feature affordances while preserving the user goal. For website V3, R1--R4 denote successive reflection-rule sets obtained through iterative reflection on observed failures and rule revision.

\subsection{\expel{}-Style Rules Help, but Do Not Fully Transfer}

On \linkding{} V1.45.0, \expel{}-style rules provide a strong training-task gain: success increases from \(13.2\%\) without rules to \(82.4\%\) with retrieved \expel{}-style rules. This confirms that reusable experience is a meaningful starting point rather than a weak baseline. The same rules, however, do not uniformly improve held-out behavior. On held-out validation tasks from \linkding{} V1.45.0, success changes from \(79.6\%\) without rules to \(74.3\%\) with \expel{}-style rules. This motivates separating training tasks from held-out validation tasks instead of reporting a single averaged score.

\subsection{Reflection Rules Help Under Moderate Website Drift}

We next test whether reflection rules add value when the website changes but the task goals remain the same. On website V2, adding reflection rules improves the training tasks from \(88.2\%\) to \(98.5\%\), and improves the held-out validation tasks from \(68.3\%\) to \(85.6\%\) (Table~\ref{tab:website_version_summary} and Figure~\ref{fig:website_version_overview}). Since the model, tasks, website version, and \expel{}-style rules are held fixed, these gains isolate the benefit of rules that describe how to preserve intent across interface changes.

\subsection{Website V3 Exposes the Robustness Gap}

The harder generated website suite V3 creates the main robustness stress test. With \expel{}-style rules alone, success drops to \(11.8\%\) on the training tasks. The strongest reflection-rule set recovers performance to \(95.6\%\), while three alternative rule sets reach \(88.2\%\) (Table~\ref{tab:v3_reflection_overall}). Figure~\ref{fig:v3_drift_rule_profile} shows that the recovery is broad across drift families rather than coming from a single easy bucket. This large gap shows that robustness under website drift is not obtained simply by adding more task-level guidance; the useful rules are those that explicitly describe how to recover from changed navigation paths, delayed readiness, changed labels, and shifted workflow cues.

The held-out validation tasks remain substantially harder. \expel{}-style rules alone reach \(39.5\%\). The two strongest reflection-rule sets, R1 and R4, reach \(58.1\%\), while weaker variants R3 and R2 reach \(49.1\%\) and \(36.5\%\). Thus, reflection rules improve transfer under harder drift, but the gains are uneven: a plausible rule update can improve one family of failures while regressing another.

\subsection{Rule Updates Are Useful but Non-Monotonic}

The detailed website V3 matrices in Appendix Tables~\ref{tab:training_v3_appendix} and~\ref{tab:heldout_v3_appendix}, summarized visually in Figure~\ref{fig:v3_reflection_iteration} and Figure~\ref{fig:v3_drift_rule_profile}, show that the best aggregate reflection-rule set is not simply the most recent edit. One evidence-based update improves several held-out drift categories, including authentication-related, visual, content, and timing changes, but loses ground on workflow and feature-affordance changes. The aggregate tie on the held-out validation tasks therefore hides a real tradeoff between different recovery behaviors.

This pattern supports the main design choice in \webcoevo{}: rule generation, verification, and promotion should be separate stages. Matched trajectories can suggest useful edits, but an edit should not be promoted only because it is plausible or locally evidence-grounded. It must also preserve previously successful behaviors under the same website-drift distribution.

\subsection{Failure Analysis: Why Iterative Reflection Degrades Held-Out Generalization}
\label{sec:failure_analysis}

The per-drift breakdown in Appendix Tables~\ref{tab:training_v3_appendix} and~\ref{tab:heldout_v3_appendix} exposes a sharp asymmetry between in-distribution repair and held-out transfer. On the 68 training tasks, successive rule sets differ modestly: R1 reaches \(65/68\), while R2--R4 each reach \(60/68\). On the 167 held-out tasks, the same iteration path is much less stable: R1 reaches \(97/167=58.1\%\), R2 falls to \(61/167=36.5\%\), and R3--R4 recover to \(82/167=49.1\%\) and \(97/167=58.1\%\). The R1$\to$R2 transition loses only five training tasks but thirty-six held-out tasks, an approximately \(7{\times}\) larger loss on the transfer split. This gap suggests that the main failure is not weak optimization on the observed traces, but a mismatch between the local evidence used to edit rules and the broader conditions under which those rules are later applied.

\paragraph{Reflection edits trade specificity for coverage.}
Across R1--R4, the strongest rules are not the most detailed ones; they are the ones that preserve abstract completion evidence across many website states. R1 succeeds because several rules describe invariants such as destination preservation, query-state evidence, and visible target confirmation. R2, by contrast, often replaces broad stopping conditions with narrower procedural advice targeted at the most recent training failures. These edits are locally rational: they explain why the agent failed on the observed trajectory. They are also globally risky: when a rule changes from ``what state proves completion'' to ``what action to take at this checkpoint,'' it no longer protects held-out tasks whose surface workflow differs but whose completion evidence is already sufficient. The resulting degradation is concentrated in \texttt{process} and \texttt{functional} tasks, where agents must decide when a multi-step interaction has already satisfied the goal rather than merely find the next clickable element.

\paragraph{Observable failures recover faster than semantic failures.}
Later iterations partially repair categories whose errors leave direct page-level evidence. Access and surface drift provide visible signals such as a wrong redirect, missing entry point, or changed navigation affordance; these signals are easy for a trajectory-to-rule editor to identify and rephrase. This helps explain why R3 and R4 improve the held-out \texttt{access}, \texttt{surface}, and \texttt{runtime} buckets relative to R2. The same process is weaker for \texttt{process}, \texttt{functional}, and \texttt{structural} drift. In these cases, success depends on an abstract relation between goal state, route state, and task semantics. Once an iteration-specific patch displaces that abstraction, subsequent trace evidence may show repeated failure but not reveal the missing principle unless the prompt explicitly asks the generator to preserve previously successful stopping behavior.

\paragraph{Content drift reveals a different limitation.}
The \texttt{content} bucket is the only held-out drift family where \expel{}-style rules remain stronger than every reflection-augmented rulebook: \expel{} reaches \(11/14=78.6\%\), while R1--R4 range from \(5/14\) to \(10/14\). This is not the same failure as the R1$\to$R2 process collapse. Content drift changes labels, placeholders, and nearby wording while keeping the data model intact, so robustness requires tolerance to lexical variation. The \expel{}-style layer, distilled from broader task experience, appears to encode this tolerance as a diffuse behavioral prior. The reflection layer is more brittle because its evidence comes from a narrower set of generated content variants; even when a rule is written in general language, its implicit anchors can remain tied to the strings and form contexts seen during mining.

\paragraph{Implication for reflection-rule learning.}
The results point to a generalization problem in the rule editor rather than a failure of reflection as a mechanism. Reflection improves V3 robustness substantially, but unconstrained iteration can convert transferable principles into high-precision patches. A safer promotion criterion should therefore evaluate three properties separately: whether an edit explains the new failure, whether it preserves the old success conditions that the replaced rule covered, and whether its predicates are expressed in observable state evidence rather than task-specific surface tokens. In other words, the reflection loop should treat rule updates as policy changes with regression risk, not as monotonic memory accumulation. Appendix~\ref{sec:case_studies} provides the trace-level case studies and rule diffs that ground these mechanisms.
